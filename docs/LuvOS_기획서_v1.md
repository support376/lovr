# LuvOS 제품 기획서 v1.0

> **한 줄 정의**: 나와 상대들의 모든 데이터를 축적해 연애를 전략적으로 운영하는 개인 데이팅 OS.
>
> **본질**: Rizz는 답장 자판기. LuvOS는 관계 운영체제.
>
> **작성일**: 2026.04.19 · **작성**: Circle21
> **Status**: v1 Draft — MVP 스펙 확정용

---

## 1. 제품 정의

### 1.1. Elevator Pitch (30초)

"당신이 지금 만나고 있는 5명의 상대. 각자 어떤 사람이고, 어디까지 와 있고, 다음에 뭘 해야 하는지 — 기억할 필요 없음. LuvOS가 기억하고, 분석하고, 지시함. 데이트 중에도 실시간으로."

### 1.2. 핵심 키워드

- **Stateful** — 한 번 쓰고 잊히는 게 아니라 쌓이는 시스템
- **Multi-target** — 동시 여러 명 관리
- **Learning** — 내 성공/실패 패턴을 학습해 개인화
- **Real-time** — 데이트 중 실시간 전략 제공
- **Mobile-first** — MZ가 쓰는 디바이스에서

### 1.3. Brand Tagline 후보

- "연애도 팀플이다. 너의 참모를 고용하라."
- "Your personal dating intelligence."
- "기억은 내가, 연애는 네가."
- "연애 운영체제."

## 2. 문제 정의

### 2.1. MZ 데이팅의 실제 Pain

1. **동시 다발적 매칭** — 틴더/범블/재미 등 3-5개 앱 병행. 5-10명 동시 대화. **누가 누군지 헷갈림.**
2. **맥락 손실** — 지난번에 뭔 얘기 했는지 기억 안 남. 답장 때마다 스크롤업.
3. **전략 부재** — 답장 한 줄은 쥐어짜도, 관계 전체 방향성은 없음.
4. **자기 패턴 인식 불가** — 왜 매번 3주차에 멀어지는지 본인도 모름.
5. **데이트 당일 긴장** — 화장실에서 뭘 확인해야 할지 모름.
6. **성공/실패 복기 안 됨** — 관계 끝나면 데이터가 증발.

### 2.2. 기존 솔루션의 한계

| 도구 | 역할 | 한계 |
|---|---|---|
| Rizz, DatingAI | 답장 생성 | Stateless. 1회성. 전략 없음. |
| Hinge AI | 프로필 피드백 | 플랫폼 종속. 대화 도움 안 됨. |
| 재미있는 사람 되기 YouTube | 이론 학습 | 내 상황에 안 맞음. 실행 안 됨. |
| 친구 상담 | 맥락 공유 | 귀찮음. 친구도 기억 못 함. 편향됨. |
| 노션 일기 | 기록 | 분석/추천 없음. 결국 안 씀. |

**공통 공백**: "축적된 맥락 위에서 작동하는 개인화된 전략 엔진"은 아무도 안 만듬.

## 3. 타겟 사용자

### 3.1. 1차 타겟

**"데이팅 앱 적극 유저 중 주 2회 이상 대화하는 MZ"**

- 연령: **24-33세** (코어 26-30)
- 성별: **초기 7:3 남/여**, V2에서 여성 비중 확대
- 위치: **서울/수도권 우선**
- 직업: 사무직, IT, 마케팅, 자영업 (가처분 소득 $30K+ USD equiv)
- 디바이스: **모바일 100%** (PC는 보조)
- 행태:
  - 틴더/범블/위피/글램/데이트잇/재미 등 2개+ 앱 병행
  - 월 5-15회 매칭
  - 실제 만남 월 1-3회
  - 진지한 관계 찾는 중, 근데 여러 명 동시 탐색

### 3.2. Persona 1 — "동시 다발형 지민 (28, 남, IT)"

```
- 틴더 + 범블 + 위피. 동시 7명 대화 중.
- 월요일 아침에 수요일 만날 사람 이름 헷갈려서 실수.
- "저번에 뭐라고 했더라..." 스크롤업 하루 30분.
- 데이트 성공률 체감 30%. 왜 안 됐는지 모름.
- Pain: 기억 관리, 전략 부재.
- LuvOS에 기대: "이 사람 dossier 열어 → 브리핑 확인 → 나감"
```

### 3.3. Persona 2 — "신중한 유진 (30, 여, 마케팅)"

```
- 진지한 관계 원함. 근데 속지 않으려 관찰 중.
- 동시에 2-3명과 썸. 한 명 고르려 고민.
- 각 상대의 진정성, 애착 유형, 장기 호환성 알고 싶음.
- Pain: 사람 보는 눈 + 내 감정 정리.
- LuvOS에 기대: "이 사람은 어떤 유형인지 분석. 내가 흔들리는 지점 알고 싶음."
```

## 4. 핵심 차별화

### 4.1. Rizz vs LuvOS 비교표

| 축 | Rizz | **LuvOS** |
|---|---|---|
| **상태** | Stateless (1회성) | **Stateful (축적형)** |
| **상대 수** | 각 스크린샷 독립 | **다중 상대 dossier 관리** |
| **내 프로필** | 없음 | **15축 심리 프로파일 (기존 자산)** |
| **시간축** | 없음 | **인터랙션 타임라인** |
| **전략** | 답장 한 줄 | **다음 3-4 수 + 관계 단계 판단** |
| **실시간** | 텍스트만 | **오프라인 데이트 음성 입력** |
| **학습** | 암묵적 | **명시적 라벨링 + 개인 playbook** |
| **출력** | 짧은 메시지 | **브리핑 / 전략 / 메시지 / 디브리핑** |

### 4.2. 방어 가능한 해자

1. **데이터 플라이휠** — 유저가 쓸수록 개인 playbook이 쌓임. 이걸 두고 이탈 못 함.
2. **Self 심리 프로파일** — luvOS 15축 (Schwartz, Big Five, Attachment, Moral Foundations, Gottman) 기존 자산 재활용.
3. **3-tier LLM 아키텍처** — Closr에서 검증된 Haiku/Sonnet/Opus 분업. 실시간/전략/심층 분석 동시 처리.
4. **당사자 녹음 기반 실시간 모드** — Rizz가 법적/정책 리스크 때문에 못 건드리는 영역. 클로바 노트 선례로 이미 합법 카테고리.

## 5. 제품 구조 (5개 레이어)

### 5.1. Layer 1 — Entity Layer

두 종류의 Entity가 시스템의 근간.

**Self Entity (나)**
- 15축 심리 프로파일 (온보딩 15분 퀴즈)
- 대화 스타일 샘플 (내 기존 메시지 학습)
- 과거 패턴 ("나는 X타입에게 Y전략이 잘 먹혔다" 누적)
- 목표 (캐주얼 / 진지 / 결혼)
- 현재 가용 리소스 (주간 만남 가능 횟수, 예산)

**Target Entity (상대별 dossier)** — 여러 개 동시 관리
- 기본 프로필 (사진, 이름, 나이, 직업, 매칭 플랫폼, 관심사)
- 심리 신호 (대화에서 LLM이 추론한 가치관/애착 유형/커뮤니케이션 스타일)
- 행동 패턴 (답장 속도, 대화 주도권, 이모지 사용, 약속 잡기 주체)
- 관계 단계 (매칭 → 탐색 → 썸 → 호감 확정 → 관계 진입 / 또는 소강/종료)
- 주요 지뢰 / 강점 (LLM이 자동 추출)
- 태그 / 메모 (유저 자유 입력)

**Entity 간 관계**
- 같은 친구 풀인지 (회사/동네 겹침 → 주의)
- 매칭 플랫폼 (틴더 매칭 vs 소개팅 → 전략 다름)

### 5.2. Layer 2 — Interaction Timeline

각 Target별 시계열 이벤트. 팔란티어의 타임라인 뷰와 동일 구조.

**이벤트 타입**
- `MATCH` — 매칭 발생
- `MESSAGE_SENT` / `MESSAGE_RECEIVED` — 메시지 교환 (스크린샷 자동 파싱)
- `DATE` — 오프라인 만남 (장소, 시간, 분위기, 음성 녹음 선택)
- `STATUS_CHANGE` — 관계 단계 변동
- `NOTE` — 유저 자유 메모
- `STRATEGY_USED` — 추천 전략 채택 기록
- `OUTCOME_LABEL` — 결과 라벨 (잘됨/망함/모호) + 원인 태그

**자동 추출 필드**
- 각 메시지의 감정 톤
- 응답 시간 간격 변화 (관심도 지표)
- 주제 분포 변화 (대화 깊이)

### 5.3. Layer 3 — Strategy Engine

현재 상태 + 과거 데이터 + Self 프로파일 + Target 프로파일 → 다음 수 추천.

**출력 타입**
- **Situation Report**: "이 관계는 지금 썸 단계. 상대 반응 주도권 70%. 3일간 메시지 톤 평탄화. 추가 진전 위험 신호."
- **Next Move Options (3-4개)**: 각 옵션에 risk/reward + 예상 결과
- **Message Draft**: 옵션 선택 시 바로 메시지 초안 생성
- **Milestone Timeline**: "2주 내 고백 단계 가려면 주차별 어떻게 해야 함"

**작동 원리**
- Target dossier + 최근 5개 인터랙션 + Self 프로파일 → Sonnet 호출
- Opus는 Post-date 전체 관계 분석 (무거운 추론)

### 5.4. Layer 4 — Real-time Assist

데이트 현장에서의 3가지 모드.

**모드 A — Pre-date Briefing** (데이트 30분 전 자동 알림)
```
▸ 오늘 상대: 박○○ (5번째 만남)
▸ 목표: 관계 확정 단계 제안
▸ 꼭 다룰 주제 3:
  1. 가족 관계 (지난번 궁금해했음)
  2. 여행 계획 (공통 관심사, 현재 모멘텀 구간)
  3. 다음 주 주말 스케줄
▸ 피할 주제: 전 연애 (지난번 급격히 식음)
▸ 이 사람 애착 유형: 회피형 경향
  → 너무 빠른 확정 요구 금물
  → 자율성 존중 신호 보내기
```

**모드 B — In-date Real-time** (데이트 중, 옵션 활성화)
- 백그라운드 녹음 (**당사자 녹음, 한국 법적 합법 / 단, 약관에 유저 고지 책임 명시**)
- 온디바이스 STT (Whisper.cpp) — 상대방 음성 원본 서버 전송 안 함
- 화자 분리: 내 발화와 상대 발화 구분
- 푸시 알림 조건부 on/off (진동만, 예: 대화 20분간 네거티브 톤 지속 시)
- **화장실 체크인 모드**: 앱 열면 지금까지 대화 3줄 요약 + 다음 30분 제안

**모드 C — Post-date Debriefing** (데이트 종료 직후)
- 전체 녹음 분석 → dossier 자동 업데이트
- 주제별 상대 반응 곡선 (감정 그래프)
- "다음 약속 타이밍 추천": 24시간 내 메시지, 48시간 후 다음 제안
- 내가 잘한 점 / 개선점 3가지

### 5.5. Layer 5 — Learning Loop

개인 playbook이 쌓이는 레이어.

**명시적 학습**
- 데이트/관계 종료 시 유저가 결과 라벨링 ("성공 / 실패 / 애매")
- 원인 태그 선택 또는 자유 기술
- LLM이 "이 유저는 X타입 상대에게 Y전략이 평균 Z% 성공" 집계

**암묵적 학습**
- 어떤 추천을 채택했고 안 했는지
- 어떤 톤의 메시지를 선호하는지

**집단 학습 (Privacy-preserving)**
- 익명화된 집계: "26세 남성 INFP가 ENFP 여성에게 평균 4주차에 다음 단계 제안 시 성공률 N%"
- k-익명성 보장 (n≥100 이상 집계만 공개)

## 6. 핵심 사용자 여정 (모바일 UX)

### 6.1. 온보딩 (최초 15분)

1. **Welcome** — "너의 연애를 운영체제로 만든다"
2. **심리 프로파일** — 15축 퀴즈 (게이미피케이션, 5분)
3. **대화 스타일 캡처** — 최근 카톡 스크린샷 3장 업로드 → 내 톤 학습
4. **현재 상대 추가** — 1-2명 바로 등록 유도
5. **첫 Strategy Report 생성** — "이게 LuvOS입니다" 경험

### 6.2. 일상 사용 (하루 평균)

**아침** — 홈 화면 열면 각 Target별 상태 카드
```
━━━━━━━━━━━━━━━
박○○ · 썸 단계
마지막 대화 8시간 전
⚠ 답장 타이밍 지연 중
→ 탭해서 조치
━━━━━━━━━━━━━━━
김○○ · 탐색 단계
데이트 예정: 내일 19시
→ 브리핑 보기
━━━━━━━━━━━━━━━
```

**대화 중** — 스크린샷 업로드 or 텍스트 붙여넣기
- 답장 옵션 3개 (각자 다른 전략/톤)
- "이 답장을 쓰면 상대 반응 예측" 표시

**데이트 전** — 푸시로 브리핑 자동 전송

**데이트 중** — 위젯에서 녹음 on/off, 화장실 체크인

**데이트 후** — 앱이 먼저 물음: "오늘 어땠어? 결과 간단 기록"

### 6.3. 주간 Review

- 일요일 밤 푸시: "이번 주 너의 연애 리포트"
- 각 Target 진척도
- 이번 주 잘한 점, 패턴 발견, 다음 주 초점

## 8. 기술 아키텍처

### 8.1. 클라이언트

- **Stack**: Next.js + PWA (iOS/Android 앱스토어 우회, 모바일 네이티브 feel)
- **오디오**: WebRTC + 네가 이미 만든 수음 모듈 이관
- **온디바이스 STT**: Whisper.cpp (WASM) — 상대방 음성 원본 서버 전송 안 함
- **오프라인 캐싱**: IndexedDB — 데이트 중 네트워크 불안정해도 기록 가능

### 8.2. 서버

- **API**: FastAPI or Node + tRPC
- **DB**: PostgreSQL (Entity + Timeline) + Redis (실시간 상태) + Vector DB (의미 검색)
- **LLM 3-tier** (Closr와 공유 코드베이스):
  - **Haiku** — 실시간 체크인 (5초 응답)
  - **Sonnet** — 전략 수립 (5분 이내)
  - **Opus** — 사후 디브리핑, 주간 리포트
- **Prompt caching** — Target dossier 큰 컨텍스트는 캐싱해 비용 70%+ 절감
- **STT (보조)**: Naver Clova (한국어) — 온디바이스 실패 시 폴백

### 8.3. 개인정보 / 보안

- 상대방 음성 원본은 **서버 전송 안 함** (온디바이스 STT 후 텍스트만)
- 상대방 식별 정보는 유저가 alias로만 입력 가능 (실명 선택)
- Target 정보는 처리위탁 구조 (Anthropic/OpenAI API 호출은 Circle21 업무 처리 위탁)
- 개인정보 처리방침에 수탁자 공개
- 모든 데이터 AES-256 at rest, TLS 1.3 in transit
- 유저는 언제든 특정 Target 완전 삭제 가능

### 8.4. Closr와의 코드 공유

| 모듈 | Closr | LuvOS | 재사용 |
|---|---|---|---|
| 수음/오디오 | ✓ | ✓ | 100% |
| 3-tier LLM 파이프라인 | ✓ | ✓ | 100% |
| Prompt caching | ✓ | ✓ | 100% |
| STT 엔진 | Clova (법률/의료) | Deepgram/Whisper (대화) | 50% |
| 엔티티 DB | 고객 | 상대 | 스키마 유사 |
| 프롬프트 템플릿 | 영업 코칭 | 연애 전략 | 다름 |

**개발 단축**: Closr 스택 기반 2-3주 단축.

## 13. 성공 지표 (KPI)

### 13.1. North Star Metric

**"Weekly Active Relationships (WAR)"** = 주간 활성 상대 수 × 유저 수

이게 늘수록: 유저가 쓰고 있고, 여러 명 관리 중이고, 실제 연애 중.

### 13.2. 단계별 KPI

**베타 (0-3개월)**
- WAU / MAU > 50%
- 평균 Target 수 ≥ 2개
- 주간 대화 기록 건수 ≥ 5건/유저
- NPS ≥ 40

**성장 (3-12개월)**
- 유료 전환율 ≥ 8%
- MRR 성장 ≥ 30%/월
- CAC < LTV/3
- 관계 성공 신고율 (자발적) 측정 개시

**스케일 (12개월+)**
- MAU 100,000+
- ARR 10억원+
- 한국 외 확장 타진 (일본/동남아)

```
LuvOS (개인 연애 OS)
  ↓ 전환 funnel
re:Heart (실제 만남 이벤트)
  ↓ 프리미엄 업셀
re:Heart Black (1:1 매칭)
```

LuvOS 유저 중 "실제 사람 만나는 이벤트"에 관심 있는 층은 re:Heart로 자연 유입. 반대도 가능.

### 14.3. 제품 경계

- LuvOS = **개인 도구** (혼자 씀)
- re:Heart = **오프라인 매칭 서비스** (사람이 운영)
- 둘은 분리된 브랜드, 데이터 공유 안 함, 단 유저가 명시적 연결 시 크로스 추천
